# Paper — MSQA-Bench

This package implements **MSQA-Bench**: dataset curation from **existing** Q&A (e.g. already generated by your QA pipeline), human-annotation sampling, **retrieval** evaluation, and paper tables.

**Important:** The orchestrated pipeline (`scripts/run_paper_pipeline.py`) **does not** call an LLM to regenerate questions or answers. It reads `data/consolidated_qa.jsonl` and enriches, splits, and measures retrieval over that data.

## Directory Structure

```
paper/
├── annotation/          # Human evaluation tools
│   └── gold_set_sampler.py
├── dataset/             # Dataset engineering pipeline
│   ├── metadata_extractor.py
│   ├── quality_filters.py
│   ├── question_classifier.py
│   ├── schema_enricher.py
│   └── split_generator.py
├── evaluation/          # Retrieval metrics
│   └── retrieval_baselines.py
└── figures/             # Paper tables and visualizations
    └── generate_tables.py
```

## Modules

### 1. Dataset (`dataset/`)

Transforms consolidated QA pairs into benchmark-quality records:

- **`metadata_extractor.py`** — Extracts document metadata (DOI, PMID, arXiv ID, title, year, venue, license) from source text using regex patterns and optional CrossRef API enrichment.
- **`quality_filters.py`** — Filters and deduplicates QA pairs. Computes quality metrics including answer-context overlap, question specificity, answer completeness, and MinHash-based deduplication (LSH).
- **`question_classifier.py`** — Rule-based classification of questions into 7 types: factual, method, definition, comparison, numeric, causal, and unknown. Includes MS-domain-specific indicators.
- **`schema_enricher.py`** — Enriches each QA record with metadata, evidence spans, question type, answer style (extractive/abstractive), quality score. Produces the `EnrichedQARecord` schema. Includes `to_public_dict()` for safe release.
- **`split_generator.py`** — Deterministic document-level train/val/test splitting (85/10/5) using MD5 hashing. Ensures all QA pairs from the same paper stay in the same split to prevent data leakage.

### 2. Annotation (`annotation/`)

- **`gold_set_sampler.py`** — Stratified sampling of QA pairs for human evaluation. Exports annotation-ready CSV/JSONL with fields for answer correctness, evidence support, evidence quality, and question clarity. Computes inter-annotator agreement (Cohen's Kappa).

### 3. Evaluation (`evaluation/`)

- **`retrieval_baselines.py`** — BM25 (sparse) and embedding-based (dense) retrieval. Metrics: Recall@k, MRR, NDCG, MAP. Evaluates whether the **correct context** can be retrieved for each question (your stored `question` / `answer` / `context` are unchanged).

### 4. Figures (`figures/`)

- **`generate_tables.py`** — LaTeX tables: dataset statistics, question type distribution, retrieval comparison.

## Pipeline Flow

```
Consolidated QA (data/consolidated_qa.jsonl)  ← your existing Q&A
  │
  ├─► quality_filters        → filter low-quality, deduplicate
  ├─► question_classifier    → factual / method / definition / ...
  ├─► schema_enricher        → metadata, evidence spans, answer style
  ├─► split_generator        → train / val / test (document-level)
  │
  └─► Enriched QA (paper_results/dataset/)
        │
        ├─► gold_set_sampler      → sample for human annotation
        ├─► retrieval_baselines   → retrieval metrics (no new answers)
        └─► generate_tables       → LaTeX tables for paper
```

## How to Run

### Prerequisites

- Input: `data/consolidated_qa.jsonl` (from `scripts/consolidate_qa.py` or equivalent)
- Optional: `data/extracted_text/bulk_40k/` (better metadata from full text; falls back to `context` in each row)
- Activate env: `source .venv/bin/activate`

### Full Pipeline

```bash
python scripts/run_paper_pipeline.py --config config/paper_pipeline.json

python scripts/run_paper_pipeline.py --config config/paper_pipeline.json --steps dataset
python scripts/run_paper_pipeline.py --config config/paper_pipeline.json --steps evaluation
```

### Run in Background (Server)

```bash
./scripts/run_paper_pipeline_background.sh
tail -f paper_results/pipeline_background.log
kill $(cat paper_results/pipeline.pid)
```

### Manual Step-by-Step

```bash
# 1. Quality filters
python -m paper.dataset.quality_filters -i data/consolidated_qa.jsonl -o paper_results/dataset/quality_filtered.jsonl

# 2. Question classifier
python -m paper.dataset.question_classifier -i paper_results/dataset/quality_filtered.jsonl -o paper_results/dataset/classified.jsonl

# 3. Schema enricher (optional: --text-dir data/extracted_text/bulk_40k)
python -m paper.dataset.schema_enricher -i paper_results/dataset/classified.jsonl -o paper_results/dataset/enriched.jsonl --text-dir data/extracted_text/bulk_40k

# 4. Split generator
python -m paper.dataset.split_generator -i paper_results/dataset/enriched.jsonl -o paper_results/dataset/splits/

# 5. Retrieval baselines
python -m paper.evaluation.retrieval_baselines -i paper_results/dataset/enriched.jsonl -o paper_results/evaluation/retrieval_results.json --compare --sample 500

# 6. Gold set for annotation
python -m paper.annotation.gold_set_sampler -i paper_results/dataset/enriched.jsonl -o paper_results/annotation/gold_set_sample.csv -n 200

# 7. Generate LaTeX tables
python -m paper.figures.generate_tables -q paper_results/dataset/enriched.jsonl -o paper_results/figures/ -r paper_results/evaluation/retrieval_results.json
```

### Parallel processing (large JSONL, e.g. 1.2M pairs)

Dataset steps use **chunked multiprocessing** when `parallel.dataset_workers` is greater than 1:

| Module | What runs in parallel | CLI flags |
|--------|------------------------|-----------|
| `quality_filters` | Metric scoring per line (MinHash dedup stays sequential in main process) | `--workers`, `--chunk-size` |
| `question_classifier` | Rule-based classification per line | `--workers`, `--chunk-size` |
| `schema_enricher` | Enrichment per line (no `--metadata` / `--public` with workers) | `--workers`, `--chunk-size` |

In **`config/paper_pipeline.json`**, set e.g. `"dataset_workers": 8` under `parallel`, or override at runtime:

```bash
python scripts/run_paper_pipeline.py --config config/paper_pipeline.json --workers 8
```

**Retrieval (GPU):** embedding models use **`evaluation.retrieval.device`** (`"auto"` → CUDA if available) and **`embed_batch_size`** (default raised in config to 128). BM25 is CPU-only.

### Configuration

Edit `config/paper_pipeline.json` for paths (`input.qa_file`, `input.text_dir`, `input.text_dir_fallbacks`), output `base_dir`, `parallel.*`, retrieval sample size / device / batch size, etc.

## Key Design Decisions

- **Document-level splits** prevent data leakage — all QA pairs from the same paper stay together.
- **Deterministic MD5 hashing** ensures reproducible splits regardless of processing order.
- **Quality scoring** weights: overlap (0.3), specificity (0.25), completeness (0.2), length (0.2).
- **Public-safe export** via `to_public_dict()` omits copyrighted context, retaining only citation offsets.
